# learnings.kb — durable insights from dev sessions

A learning is a non-obvious fact about the codebase, design space, or
process that a future agent should know but couldn't easily rederive.
Distinguished from a claim by the fact that it's about **us** (our
code, our patterns, our process) rather than external behavior.

Belongs here:
- Pattern observations across the codebase
- Field-shape comparisons between data sources
- Refactor signals ("this is where the abstraction will go")
- Process insights that survived a review pass

Does NOT belong here:
- Cross-cutting normative guidance → `../technical-policy.kb/`
- Component-level design rationale → component `design.kb/`
- Claims about external services or tools → component `dev.kb/claims.kb/`

Each file is one learning. Prose-only; no schema. If frontmatter
pressure emerges later, propose a schema then.
